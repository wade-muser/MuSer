Abstract

MuSeR (Music Smart Enhancer) is a cross-platform application developed for music lovers that intend to have a centralized point of playlists management. The underlying mechanisms of the application are smartly interconnecting the entities (artists, songs, albums, playlists etc.) using the principles of semantic Web.
The main shining features of MuSeR are: easy interlinked search through artists, songs, albums, genres and playlists, lookup on different events related to the artists, create and manage playlists manually or generate "smart" playlists through a series of keywords and the existing created playlists.

Architecture

'serverless'-side

The two main paradigms used in the design of MuSeR are 'serverless' and 'microservices'.

'serverless' states that the functionalities of the application should not depend on any server entity and they should run unknowingly 'anywhere' (e.g. cloud). The Node module 'serverless' toolkit helps us describe and deploy the mechanisms of our application in the AWS Lambda, the cloud service used to 'host' the main workhorses.

Microservices architectural style represents a distinctive method of conceiving software systems. Each functionality or service is independent, deployable, small and modular, meaning that each one is as decoupled as possible from another. Combining this concept with the 'serverless' concept, each microservice in our application runs in a separate AWS Lambda unified by a central API Gateway. This one respects the REST API concepts and it will expose different paths for each microservice:

  • PlaylistService - the main, most important service of MuSeR; it exposes the main creating, reading, updating and deleting playlists endpoints by using and aggregating results from other microservices

  • ArtistService - the service used to fetch information about artists, their records, biography and other details

  • SongService - the service that finds recordings and details about their creators and albums containing them

  • AlbumService - the service that searches collections created by artists and that contain songs

  • GenreService - the service that finds genres entities

  • EventService - the service used to discover nearby events related to music

An OpenAPI specification has been developed that describes the API paths and behaviour as you can see here: MuSeR Swaggerhub - OpenAPI. Each microservice interrogates our MuSeR RDF repository - that was pre-populated with data from DBPedia and Spotify - to find information, entities and relationships between entities that will be later used to display smart connections or to create playlists.

Authentication component relies on the JSON Web Token technology that is used by an 'authorizer' represented by an AWS Lambda function. Each call of the REST API exposed by MuSeR requires the auth token and the access is forbidden if it is invalid or missing.

Client-side

The User Interface of the application is created with the help of - maybe - the most popular cross-platform 'web-technologies' framework: Ionic, a fully fledged framework having Angular as the foundation client-side framework and Cordova as the main development environment which supports mobile devices and plugins for a whole suite of services and features.

Design Details

General considerations

  • Being a 'microservices-serverless-oriented' application, each functionality dwells into a AWS Lambda. Each of it will interrogate our central RDF MuSeR repository that holds data about all the entities and the relationships between them.

  • The user must register and generate a JWT before using the REST API exposed by MuSeR. Every subsequent call of the API will require the token.

  • The 'preferred' data transfer MIME type of MuSeR is application/json for the ease of use and the nice integration with Javascript and NodeJS.

  • Each instance's view for an entity type (e.g. an Artist or a Song) smartly suggests other instances of the same entity type taking into consideration different aspects related to it: genre, time period, preferences etc.

  • Playlists can be accessed via a SPARQL endpoint providing a way of accessing the data from the RDF repository (GraphDB).

Main mechanisms

  • 'muser_populater' is a complex mechanism created to receive a DBPedia resource entity (e.g. dbr:Pink_Floyd) and to populate the MuSeR RDF Repository with the most important data required by application's workflow; it fetches information about an artist, all the songs performed, all the albums released, all the genres related (and their derivatives) to the songs and albums; it also fetches the 'related artists' of the artist and reiterates over them

  • The RDF repository was extended by using Spotify and Songkick services developed by us; Spotify populates further the entities by fetching the images and Spotify IDs (where possible) while Songkick is used to find events for specific artists

  • Each microservice has a set of SPARQL queries that were specifically written for each entity and use case; these queries are also used to fetch the results of an API call (e.g. SongService uses find_songs_recommendations_type_album to find songs similar to the one specified by taking into account songs from the same album and it is later used by the call of the endpoint /songs/{id}/recommendations?type=album )

  • The PlaylistService exposes endpoints for creating, updating and deleting playlists. The "smart" playlists generation is a process that follows the "keyword artists" provided by the user and the songs from his personal playlists and suggest 15 random songs related to them (performed by "keyword artists" or by the artists of the songs contained by his playlists)

Use-Cases

Another aspect of the application that can be observed in the diagram is the content finding mechanism. Users - for example - can find artists based on keywords specified by them. Aside from this, the application will recommend other entities based on relationships of the visualized entity. (e.g. 'Songs from same artist', 'Songs from same Album' etc.)

Conclusion & Future considerations

MuSeR has been created to ease up the management of playlists for music enthusiasts by providing an intuitive platform that uses semantic Web concepts to enhance the way of visualizing the relationships between various entities in the context of musical domain. The application can be extended and improved in different ways by conceiving new mechanisms and architectural ideas such as media streaming or machine-learning-related suggesting mechanisms.

All of these can be integrated in the current version of the design resulting in a modern, evolved MuSeR, an user-friendly playlist manager a.k.a music enhancer.
